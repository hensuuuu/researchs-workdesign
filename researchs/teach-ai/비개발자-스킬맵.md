# 비개발자가 "hank 수준"으로 AI 쓰려면 필요한 기술 스택

> 분석 기준: 6개 실제 프로젝트 전수 분석
> 대상: F-buffer(Feedr), researchs-workdesign, ai work design, sales-item, GTD Flow, youtuboost-hook-helper

---

## 결론 먼저

hank가 하는 작업을 보면 **개발자가 아니다**. 그런데 개발자보다 많은 것을 한다.
비결은 "코딩을 배운 게 아니라, AI와 대화하는 구조를 배운 것"이다.

실제 작업 목록:
- Chrome 확장 프로그램 (youtuboost-hook-helper): TypeScript + React + Manifest V3, rate limit 처리, offscreen API
- 병원 CRM 대시보드 (sales-item): React Vite + AI 재방문 자동화, 경쟁사 분석
- GTD Flow 앱 (GTD): React 18 CDN + Supabase + Vercel, realtime partial patch, 팀 협업
- SaaS 페이크도어 (F-buffer/Feedr): 수요 검증 → 마케팅 전략 → 콘텐츠 파이프라인
- 마케팅 자동화 (ai-marketing-engine): 5-agent 파이프라인 (CMO → 인텔 → 콘텐츠 → 에디터 → 분석)

이 모든 걸 hank 혼자 한다. **코딩은 AI가, 설계/판단/실행은 hank가**.

따라서 비개발자가 같은 수준에 도달하려면 필요한 건 **코딩 지식이 아니라 아래 8개 스킬**이다.

---

## 레벨 구조 (4단계)

```
Lv.0 — 일반인 (AI 챗봇 쓰는 수준)
Lv.1 — AI 도구 사용자 (프롬프트 쓸 줄 앎)
Lv.2 — AI 설계자 (컨텍스트를 파일로 관리, 프로젝트 구조 설계)
Lv.3 — AI 오케스트레이터 (멀티 에이전트 + 자동화 파이프라인)  ← hank 수준
```

## 스킬 1: 터미널 기초 (필수, Lv.2 진입 조건)

### 왜 필요한가
hank가 하는 작업의 핵심은 **Claude Code (터미널 기반)** 사용이다.
웹 Claude와 터미널 Claude는 다르다 — 파일을 읽고 쓰고, git 커밋하고, 배포까지 한다.

### 실제로 쓰이는 명령어 (프로젝트에서 추출)
```bash
# youtuboost-hook-helper에서
npm run build
npm run build && rm -f youtuboost-hook-helper.zip && cd dist && zip -r ../youtuboost-hook-helper.zip . && cd ..

# GTD Flow에서
git add / commit / push

# 파일 탐색
find / ls / cat
```

### 배워야 할 것
- 터미널 열기 / 폴더 이동 (`cd`)
- 파일 목록 보기 (`ls`)
- git 3종 (`add`, `commit`, `push`)
- npm 기본 (`npm run build`, `npm install`)

### 학습 시간: 약 1~2일 집중 실습

## 스킬 2: 컨텍스트 파일 설계 (핵심, Lv.2 진입 조건)

### 왜 필요한가
6개 프로젝트를 보면 **모두 컨텍스트 파일 구조가 있다**.
이게 없으면 AI는 매번 처음부터 시작한다.

ai work design 문서:
> "컨텍스트 세금 — AI가 봐야 할 것을 매번 다시 알려줘야 함. 줄이려면: memory, CLAUDE.md, skill, 레퍼런스 문서 정리."

### 실제 파일 패턴 (프로젝트별)

**제품/앱 프로젝트 (GTD, feedr, youtuboost)**
```
프로젝트/
  CLAUDE.md          ← AI에게 주는 프로젝트 브리핑 + 규칙
  SPEC.md            ← 시스템 규칙 + 데이터 구조 + 스펙
  CHECKLIST.md       ← 배포 전 체크리스트
  BUGS.md            ← 과거 버그 기록 (재발 방지)
  CTX_SAVE.md        ← 세션 간 컨텍스트 저장 (youtuboost 방식)
```

**마케팅/리서치 프로젝트 (F-buffer, researchs-workdesign)**
```
프로젝트/
  context/
    기획.md            ← 제품 개요, 스택, 페이지 구성
    마케팅전략.md       ← 경쟁사 분석, 소구 포인트, 광고 카피
    브랜드메시지.md     ← 포지셔닝, 톤앤매너, 금지 표현
    키워드.md           ← SEO 키워드 목록
  _context/
    fakedoor-content-rule.md  ← 콘텐츠 생산 규칙 (나중에 agent 프롬프트로)
    fakedoor-loop.md          ← 실험 루프 구조
  logs/
    experiment-history.md     ← 실험 결과 누적
```

### 배워야 할 것
- 마크다운(.md) 파일 작성
- CLAUDE.md 구조 설계 ("이 프로젝트는 ~입니다" 형태)
- 자신의 작업 규칙을 파일로 인코딩하는 습관
- 프로젝트 성격에 따른 컨텍스트 파일 종류 선택

### 학습 시간: 약 3~5일 (프로젝트 하나 실제로 설정해보기)

## 스킬 3: 골든 예시 라이브러리 구축 (반복 세금 감면)

### 왜 필요한가
ai work design:
> "추상 규칙 10줄보다 구체 예시 2개가 압도적으로 강력하다."
> "내 브랜드·톤·취향이 박힌 결과물이어야 AI가 '나다운' 출력을 낸다."

F-buffer의 `fakedoor-content-rule.md`는 실제 이 원칙의 적용 — 채널별 포맷, 톤, 구조를 구체 예시로 명시해서 매번 설명 없이 재사용 가능하게 만들었다.

### 실제 사용 방식
- `tone_samples.md` — 내 글투 문장 10~20개
- `hook_library.md` — 반응 좋았던 훅 20~50개
- `structure_templates.md` — 잘 된 콘텐츠의 뼈대
- 채널 규칙 문서 — 네이버 블로그/X/인스타/쇼츠별 포맷, 길이, 구조, 톤 명시

### 배워야 할 것
- 내 과거 결과물 중 "잘 된 것" 분류 기준
- 골든 예시를 프롬프트에 붙이는 방법
- Taste File 유지 습관 (교정할 때마다 한 줄 기록)

### 학습 시간: 처음 세팅 30분, 이후 습관으로 누적

## 스킬 4: 파츠 사고 (아키텍처 감각, Lv.2~3 경계)

### 왜 필요한가
hank가 GTD Flow, Feedr, 병원 CRM, Chrome 확장 등을 동시에 설계하는 능력.
코딩은 AI가 하지만, **무엇을 만들지 쪼개는 것**은 hank가 한다.

sales-item DASHBOARD_PLAN.md를 보면 hank가 직접 설계한 구조:
- 탭 3개 (한눈에 보기 / 재방문 관리 / 환자 히스토리)
- 각 탭의 책임, 자동화 플로우, 화면 구성 명시
- 기술 아키텍처: `src/tabs/`, `src/components/`, `src/config/` 분리 원칙

이건 코드가 아니다. 이건 **설계**다.

### 실제로 쓰이는 방식
- 기능 나열 대신 "책임" 단위로 분해
- 파츠 간 인터페이스(뭘 주고받을지) 먼저 정의
- Walking Skeleton(뼈대 먼저, 살 나중) 방식으로 조립
- 세로 슬라이스: 한 기능이 데이터→로직→화면까지 다 돌아야 1단위

### 배워야 할 것
- "이게 하는 일이 뭐야?" 한 줄로 말하는 연습
- 기능 ≠ 파츠 구분
- 세로 슬라이스 vs 가로 슬라이스 차이
- 의존성 방향 설계 (A가 B를 알아야 하는가, B가 A를 알아야 하는가)

### 학습 시간: 약 1~2주 (설계 연습 5회 이상)

## 스킬 5: 멀티 에이전트 설계 (Lv.3, hank 수준의 핵심)

### 왜 필요한가
hank의 ai-marketing-engine:
```
CMO 에이전트 (전략 총괄)
  → 인텔 에이전트 (트렌드 수집 → JSON 저장)
  → 콘텐츠 에이전트 (초안 생성 → hank 피드백)
  → 에디터 에이전트 (검수 → final/ 저장)
  → 분석 에이전트 (성과 분석 → insights/ 업데이트)
```

GTD Flow의 CLAUDE.md도 동일 구조 — 에이전트 역할별로 분리된 명령 체계.

**사람이 하는 것:** 전략 결정, 피드백, 최종 발행
**AI가 하는 것:** 조사, 초안, 검수, 분석 — 전부

### 실제 에이전트 간 통신 방식
- 파일 기반: 한 에이전트가 JSON 저장 → 다음 에이전트가 읽어서 처리
- 에이전트별 CLAUDE.md 분리 (역할, 입력, 출력 명시)
- CMO 에이전트가 오케스트레이션 (어떤 에이전트를 언제 부를지)

### 배워야 할 것
- subagent 개념 (Claude Code 내에서 에이전트 나누는 방법)
- 에이전트 역할 정의 방법 (CLAUDE.md 분리)
- 에이전트 간 정보 전달 방식 (파일 기반)
- 오케스트레이터 vs 실행자 구조

### 학습 시간: 1~2개월 (실제 프로젝트 운영 경험 필요)

---

## 스킬 6: 이중 검토 루프 (품질 관리)

### 왜 필요한가
ai work design:
> "생성한 AI는 자기 논리에 갇혀서 빈틈을 못 봄. 같은 AI라도 새 세션에 '리뷰어' 역할 주면 다른 사람처럼 본다."

youtuboost-hook-helper는 실제로 이중 검토가 명시:
- 스펙 작성 → AI 검토 → GO/NO-GO 판정 → 구현 → 코드 리뷰

GTD IMPROVEMENT_BACKLOG.md에서: AI 3차 리뷰 통과 후 커밋 병기.

### 실제 워크플로우
```
기획 수립 (Generator 세션)
  → 기획 검토 (Reviewer 세션, 별도 대화)
  → GO/NO-GO 판정
  → 구현
  → 코드 리뷰 (/simplify + code-reviewer)
  → 배포
```

### 배워야 할 것
- "생성 세션" vs "리뷰 세션" 분리하는 습관
- 🔴/🟡/🟢 심각도 분류로 리뷰 지시하기
- 정지 규칙 ("🔴 0개면 통과")
- 사전 기획 체크리스트 7개 항목 훑기

### 학습 시간: 약 1주 (5회 이상 실습)

---

## 스킬 7: 페이크도어 / 실험 루프 설계 (비즈니스 감각)

### 왜 필요한가
hank는 제품을 만들기 전에 **페이크도어로 수요를 검증**한다.
feedr, 병원 CRM(sales-item), 마케팅 SaaS — 전부 실제 개발 전 랜딩페이지로 수요 확인.

F-buffer fakedoor-loop.md의 루프:
```
① 랜딩페이지 제작 (Claude)
② 콘텐츠 초안 (Claude)
③ 발행 (내가 직접)
④ 7-14일 대기
⑤ KPI 측정 (CTR 3%↑ 통과 / 1-3% 보류 / 1%↓ 폐기)
⑥ 판정 → 폐기 or 다음 단계
```

최대 5개 동시 실험, 예산 0원 시작 원칙.

### 배워야 할 것
- 페이크도어 개념과 설계 원칙
- 클릭 이벤트 Supabase 저장 방법 (`clicks` 테이블: product_id / plan / clicked_at)
- KPI 설정 방법 (CTR 기준, 조회수 최소 기준)
- 실험 히스토리 기록 습관

### 학습 시간: 첫 실험 설계까지 약 1주

---

## 스킬 8: OSMU 콘텐츠 파이프라인 설계

### 왜 필요한가
F-buffer의 fakedoor-content-rule.md에서 확인한 실제 구조:
```
블로그 글 1개 (Claude 생산)
        ↓
├── 쇼츠 스크립트 추출 (Claude) → 화면녹화+편집+발행 (내가)
│     └── 릴스 동시 업로드
├── X/스레드 3세트 (Claude) → 발행 (내가)
├── 인스타 카드뉴스 (Claude) → 발행 (내가)
└── 카페/오카방 요약본 (Claude) → 발행 (내가)
```

**사람이 하는 것:** 화면녹화+편집, 채널 발행 (복붙+예약)
**Claude가 하는 것:** 나머지 전부

4개 인풋 문서(기획/마케팅전략/브랜드메시지/키워드)가 없으면 콘텐츠 생산 금지 — 이게 시스템의 품질 잠금장치.

### 배워야 할 것
- 원본 서비스 서치 → 소재 추출 방법
- 채널별 포맷 규칙 (길이, 구조, 톤)
- 인풋 문서 4종 작성법
- 주간 콘텐츠 요청 형식

### 학습 시간: 약 1주 (첫 사이클 완성까지)

## 전체 로드맵 (0→hank 수준)

```
Week 1-2: 터미널 기초 + Claude Code 첫 실행
  → 터미널에서 파일 만들고, git push까지
  → Claude Code로 첫 프로젝트 파일 생성

Week 3-4: 컨텍스트 파일 설계
  → 내 프로젝트 하나에 CLAUDE.md + context/*.md 만들기
  → CTX_SAVE.md 습관 (세션 끝날 때 저장)

Week 5-6: 골든 예시 + 이중 검토 루프
  → 내 과거 결과물 정리 → tone_samples.md
  → 리뷰 세션 5회 이상 실습

Month 2: 파츠 사고 훈련
  → 새 프로젝트 설계 5회 이상 (가상이어도 됨)
  → Walking Skeleton으로 첫 기능 만들어보기

Month 2-3: 페이크도어 첫 실험
  → 아이디어 1개 → 랜딩페이지 → 콘텐츠 → KPI 측정

Month 3+: 멀티 에이전트 운영
  → 에이전트 2~3개짜리 작은 파이프라인 하나 실제 운영
  → OSMU 콘텐츠 파이프라인 자동화
```

## 비개발자가 안 배워도 되는 것

| 항목 | 이유 |
|------|------|
| React/TypeScript 문법 | AI가 다 씀. 읽을 줄만 알면 됨 |
| SQL 문법 | Supabase 대시보드 + AI로 처리 |
| CSS 상세 | AI가 디자인까지 짬. 방향만 지시 |
| 알고리즘/자료구조 | 필요 없음 |
| 프레임워크 선택 | AI가 추천, hank는 선택만 |
| Chrome Extension API | AI가 씀. `manifest.json` 구조만 이해 |
| Supabase 쿼리 작성 | 테이블 구조 설계만 하면 됨 |

---

## 핵심 한 줄

> hank 수준 = **코딩 실력이 아니라 "AI에게 뭘 줘야 하는지" 아는 능력**.
> 컨텍스트 설계 + 파츠 사고 + 멀티 에이전트 — 이 셋이 핵심이다.
> 터미널은 그 능력을 실행하는 도구이고, 페이크도어는 그 능력을 검증하는 방법이다.
